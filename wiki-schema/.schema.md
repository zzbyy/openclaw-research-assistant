# Wiki Schema

You are maintaining a personal knowledge wiki inside an Obsidian vault.
This file defines the structure, conventions, and operations you must follow.

If Obsidian skills are available (e.g., obsidian-skills in Claude Code or
openclaw-obsidian-skills in OpenClaw), use them for creating and editing
markdown files. They ensure proper Obsidian formatting.

---

## Directory Layout

```
wiki/                         ← You operate here (working directory)
├── .schema.md                ← This file (hidden from Obsidian, read by LLM)
├── .entries/                 ← Pre-extracted text from sources (read these, not raw PDFs)
├── .catalog.json             ← Source catalog (hidden, machine-readable)
├── index.md                  ← Category-organized catalog of all wiki pages
├── log.md                    ← Chronological append-only record of operations
├── concepts/                 ← Concept pages (ideas, principles, theories)
├── methods/                  ← Method/technique pages (algorithms, processes)
├── papers/                   ← Paper summary pages (specific publications)
├── people/                   ← People pages (researchers, authors)
├── books/                    ← Book summary pages
├── domains/                  ← Domain overview pages (field-level summaries)
├── tools/                    ← Tool/software pages
├── datasets/                 ← Dataset/benchmark pages
└── techniques/               ← Technique pages (practical how-to)

../sources/                   ← Raw documents (READ ONLY — never modify)
├── pdfs/
├── html/
├── epub/
└── markdown/
```

**IMPORTANT**: When ingesting, read from `.entries/` (pre-extracted text), NOT from
`../sources/` directly. The Python extraction pipeline (`ingest.py`) has already
converted PDFs/EPUBs into clean markdown text with metadata. This is faster and
more reliable than reading raw PDFs.

---

## Two-Phase Ingest Pipeline

### Phase 1: Text Extraction (Python — already done before you see the entry)
`ingest.py` extracts text from PDF/EPUB/MOBI/HTML → writes to `.entries/` as markdown.
Each entry has frontmatter: id, title, author, date, format, language, source_file.

### Phase 2: Wiki Synthesis (You — this is your job)
Read the entry from `.entries/`, then:
1. Identify key concepts, methods, people, techniques
2. Create or update wiki pages in the appropriate type subdirectory
3. Add cross-references and relationships
4. Update index.md and log.md

---

## Page Format

Pages are organized by type into subdirectories. Place each page in the right directory.

### Frontmatter (YAML)

```yaml
---
title: "Human-Readable Title"
type: concept                    # determines which subdirectory
created: 2026-04-14
last_updated: 2026-04-14
sources: ["a1b2c3d4e5f6"]       # file IDs from .entries/ frontmatter
related: ["[[other-page]]", "[[another-page]]"]
---
```

For **books** and **papers**, add author info:
```yaml
---
title: "Good Strategy Bad Strategy"
type: book
created: 2026-04-14
last_updated: 2026-04-14
sources: ["f8345264f77d"]
related: ["[[好战略的内核]]", "[[坏战略的特征]]"]
---
```

### Body

- Use Obsidian wikilinks `[[page-name]]` for all cross-references
- Use `#topic/subtopic` tags inline where relevant
- Use Obsidian callouts for special blocks:
  - `> [!info]` for key definitions
  - `> [!warning] Contradiction` for flagged contradictions
  - `> [!tip]` for practical implications
  - `> [!example]` for concrete examples
- Keep pages focused on one concept/entity — split if too long
- Use `##` headings for major sections, `###` for subsections
- End concept pages with a `## 参考` or `## References` section linking related pages

### File Naming

- Use natural names that work as wikilink targets
- Chinese titles for Chinese concepts: `好战略的内核.md`, `用户价值.md`
- English for English concepts: `Data Mesh Architecture.md`, `Growth Mindset.md`
- Book pages: add （书） suffix for Chinese, `（书）` suffix: `俞军产品方法论（书）.md`
- No kebab-case requirement — use whatever reads naturally in Obsidian

### Page Placement

| Type | Directory | When to use |
|------|-----------|-------------|
| `concept` | `concepts/` | Ideas, principles, theories, frameworks |
| `method` | `methods/` | Algorithms, architectures, specific approaches |
| `technique` | `techniques/` | Practical how-to processes and frameworks |
| `paper` | `papers/` | Summaries of specific publications |
| `book` | `books/` | Book-level summaries with links to concept pages |
| `person` | `people/` | Researchers, authors, key figures |
| `domain` | `domains/` | Field-level overview pages (e.g., "产品管理", "Immunology") |
| `tool` | `tools/` | Software, libraries, frameworks |
| `dataset` | `datasets/` | Datasets, benchmarks |

## Operations

### Absorb (Ingest Phase 2)

When asked to absorb an entry from `.entries/`:

1. **Read the entry** from `.entries/<filename>.md` — it contains extracted text with metadata
2. **Scan existing pages** — read `index.md` to understand current wiki state
3. **Identify entities** — what concepts, methods, people, techniques does this source cover?
4. **For each entity**:
   - If a page exists in the right subdirectory: update it, add the source ID, add cross-references
   - If no page exists: create a new page in the right subdirectory with full frontmatter
5. **Create a book/paper page** — one summary page for the source itself (in `books/` or `papers/`)
6. **Cross-reference** — link concept pages to the book/paper page and to each other
7. **Check for contradictions** — if new claims conflict with existing pages:
   - Add `> [!warning] Contradiction` callout with both sources cited
   - Do NOT auto-resolve — surface the conflict
8. **Update index.md** — add new pages under the appropriate categories
9. **Append to log.md** — record: date, "absorb", source entry name, pages created/updated

### Query

When asked a question against the wiki:

1. **Search relevant pages** — use index.md, type subdirectories, and wikilinks to find matches
2. **Read the pages** — gather information from matched pages
3. **Synthesize an answer** with `[[wikilinks]]` as citations
4. **Note gaps** — what the wiki doesn't cover yet
5. **Append to log.md** — record: date, "query", the question, pages referenced

### Lint

When asked to health-check the wiki:

1. **Contradiction scan** — find pages with conflicting claims
2. **Orphan detection** — pages not linked from index.md or other pages
3. **Broken wikilinks** — `[[links]]` pointing to non-existent pages
4. **Missing cross-references** — pages discussing same topics without linking
5. **Index completeness** — verify all pages across all subdirectories appear in index.md
6. **Produce a report** and append to log.md

## index.md Format

```markdown
# Wiki Index

> Master index of all wiki articles. Updated on every absorb cycle.
> **N articles** synthesized from **M sources** across K knowledge domains.

---

## Domains (K)

- [[Domain Name]] — one-line description

---

## Concepts (N)

### Topic Area
- [[Concept Name]] — one-line description

---

## Techniques (N)

- [[Technique Name]] — one-line description

---

## People (N)

- [[Person Name]] — one-line description

---

## Books (N)

| Book | Key Wiki Articles |
|------|-------------------|
| [[Book Title（书）]] | [[Concept A]], [[Concept B]] |
```

Group concepts by topic area. Alphabetical within each group.

---

## log.md Format

Append-only, narrative format:

```markdown
# Wiki Log

## [2026-04-14] absorb | paper_name.md
- 3 pages created: [[Concept A]], [[Concept B]], [[Book Title（书）]]
- 1 page updated: [[Existing Concept]] (added new source)
- No contradictions found

## [2026-04-14] query | "how does X work?"
- Referenced: [[Concept A]], [[Concept B]]
```

---

## Rules

- Read from `.entries/` (pre-extracted text), not raw source files
- Place pages in the correct type subdirectory, not a flat `pages/` directory
- Always update `index.md` and `log.md` when making changes
- Use natural filenames that work as Obsidian wikilinks
- Link source IDs (from entry frontmatter) in page `sources` field
- When updating a page, update `last_updated`
- Prefer updating existing pages over creating near-duplicates
- Use `> [!warning] Contradiction` — never silently overwrite conflicting information
